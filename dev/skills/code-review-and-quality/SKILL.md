# 代码审查与质量

## 概述

多维代码审查，带质量门禁。覆盖五维度：正确性、可读性、架构、安全、性能。

## 何时用

- 合并 PR 或变更前
- 功能实现完成后
- 其他 Agent/模型产出代码需评估
- 重构已有代码时
- bug 修复后（审修复本身 + 回归）

## 五轴审查

### 1. 正确性

代码是否做了声称的事？

- 匹配 spec/需求？
- 边界处理？（null、空值、边界值）
- 错误路径处理？（非仅 happy path）
- 测试通过？测试在测正确东西？
- 差一错误、竞态条件、状态不一致？

### 2. 可读性与简洁性

其他工程师（或 Agent）不靠解释能理解吗？

- 命名描述性且一致？（不用 `temp`、`data`、`result`）
- 控制流直观？（避嵌套三元、深层回调）
- 代码有序？（相关分组、清晰边界）
- 有可简化的"聪明"技巧？
- **能用更少行数完成？**（100 行够却写 1000 行就是失败）
- **抽象物有所值？**（第三个用例前不泛化）
- 注释澄清非显而易见意图？（不注释显而易见的）
- 死代码残留？（空操作变量、向后兼容 shim、`// removed` 注释）

### 3. 架构

变更适合系统设计吗？

- 遵循已有模式还是新模式？新的是否有理由？
- 模块边界干净？
- 有可共享的重复代码？
- 依赖方向正确？（无循环依赖）
- 抽象层级合适？（不过度工程、不太耦合）

### 4. 安全性

详安全指导见 `dev:security-and-hardening`。变更引入漏洞？

- 用户输入验证和清理？
- 密钥不暴露在代码/日志/版本控制中？
- 认证/授权在需要处检查？
- 查询参数化？（无字符串拼接）
- 输出编码防注入？
- 依赖来自可信来源且无已知漏洞？
- 外部数据视为不信任？（API、日志、用户内容、配置）

### 5. 性能

变更引入性能问题？

- N+1 查询？
- 无界循环或无约束数据抓取？
- 应是异步的同步操作？
- 列表端点缺分页？
- 热路径中创建大对象？

## 变更大小

```
~100 行    → 优秀，一次审完
~300 行    → 可接受，单逻辑变更
~1000 行   → 太大，拆分
```

**"一个变更"：** 自包含，做一件事，含相关测试，提交后系统可用。

**拆分策略：**

| 策略 | 方法 | 场景 |
|------|------|------|
| 堆叠 | 提交小变更，基于它开始下一个 | 串行依赖 |
| 按文件组 | 按需不同审查者分出独立变更 | 跨领域 |
| 横向 | 先创建共享代码/存根，再消费者 | 分层架构 |
| 纵向 | 拆更小全栈切片 | 功能开发 |

**大变更可接受：** 文件删除和自动化重构——审意图非每个行。

**重构与功能分开。** 改已有+加新行为是两个变更——分别提交。小清理（变量重命名）酌情包含。

## 变更描述

需在版本控制历史中独立存在。

**前缀：** feat、fix、refactor、docs、test、chore、perf、style。

**第一行：** 简短、祈使、独立。"删除 FizzBuzz RPC"非"正在删除"。搜索历史者无需读 diff 即理解。

**正文：** 什么在变，为什么。含代码不可见的上下文、决策和推理。缺点承认。

**反模式：** "修复 bug"、"修复构建"、"添加补丁"、"把代码从 A 移到 B"。

## 审查流程

### 第 1 步：理解上下文

看代码前，理解意图：
- 变更试图达到什么？
- 实现什么 spec/任务？
- 预期行为变更是什么？

### 第 2 步：先审测试

测试揭示意图和覆盖：
- 有此变更的测试？
- 测行为非实现细节？
- 边界覆盖？
- 名称描述性？
- 代码改变，测试能捕获回归？

### 第 3 步：审实现

五轴视角浏览，对每个文件：
1. 正确性：测试说应做的事做到？
2. 可读性：不需帮助能理解？
3. 架构：适合系统设计？
4. 安全：有漏洞？
5. 性能：有瓶颈？

### 第 4 步：分类发现

每个评论标严重级别：

| 前缀 | 含义 | 作者行为 |
|------|------|---------|
| (无) | 必须改 | 合并前处理 |
| **Critical:** | 阻止合并 | 安全漏洞、数据丢失、坏功能 |
| **Nit:** | 次要可选 | 作者可忽略 |
| **Optional:** / **Consider:** | 建议 | 值得考虑 |
| **FYI** | 参考 | 无需行动 |

### 第 5 步：快速预检

调 `dev:quick-check` 做编译检查、安全扫描、规范检查。确认无阻塞再终裁。

### 第 6 步：验证过程

检查验证叙述：
- 跑了什么测试？
- 构建通过？
- 手动测试过？
- UI 有截图？
- 前后对比？

## 审查检查清单

```markdown
## Review: [PR/变更标题]

### 上下文
- [ ] 理解变更做什么、为什么

### 正确性
- [ ] 匹配 spec/需求
- [ ] 边界处理
- [ ] 错误路径处理
- [ ] 测试充分覆盖

### 可读性
- [ ] 名称清晰一致
- [ ] 逻辑直观
- [ ] 无不必复杂度

### 架构
- [ ] 遵循已有模式
- [ ] 无不必耦合
- [ ] 合适抽象层级

### 安全性
- [ ] 代码中无密钥
- [ ] 输入在边界验证
- [ ] 无注入漏洞
- [ ] 认证检查到位
- [ ] 外部数据视为不信任

### 性能
- [ ] 无 N+1
- [ ] 无无界操作
- [ ] 列表端点有分页

### 验证
- [ ] 编译/类型检查通过
- [ ] 测试通过
- [ ] 构建成功
- [ ] 手动验证完成
- [ ] `dev:quick-check` 预检通过

### 裁决
- [ ] **批准** — 可合并
- [ ] **请求修改** — 问题须处理
```

## 参考

- 详细安全审查：`references/security-checklist.md`
- 合并前预检：`dev:quick-check`

## 警告

- PR 无审查直接合并
- 审查只检查测试通过
- 安全变更无安全聚焦审查
- 大 PR 太大无法适当审查
- Bug 修复无回归测试

## 验证

审查完成：
- [ ] 所有 Critical 已解决
- [ ] 所有 Important 已解决或明确延期
- [ ] 编译/类型检查通过
- [ ] 测试通过
- [ ] 构建成功
- [ ] 验证叙述已记录
