# 安全加固

## 概述

安全优先的开发实践。将每个外部输入视为敌对，每个密钥视为神圣，每个授权检查视为强制。安全不是一个阶段——它是触及用户数据、认证或外部系统的每一行代码上的约束。

## 何时使用

- 构建任何接受用户输入的功能
- 实现认证或授权
- 存储或传输敏感数据
- 与外部 API 或服务集成
- 添加文件上传、webhook 或回调
- 处理支付或 PII 数据

## 流程：先做威胁建模

没有威胁模型的控制措施只是猜测。在加固之前，花五分钟像攻击者一样思考：

1. **绘制信任边界。** 不信任数据从哪里进入你的系统？HTTP 请求、表单字段、文件上传、webhook、第三方 API、消息队列和 **LLM 输出**。每个边界都是攻击面。
2. **命名资产。** 什么值得窃取或破坏？凭证、PII、支付数据、管理员操作。
3. **对每个边界运行 STRIDE：**

| 威胁 | 问题 | 典型缓解措施 |
|------|------|------------|
| **S**poofing（仿冒） | 有人能冒充用户/服务吗？ | 认证、签名验证 |
| **T**ampering（篡改） | 数据在传输或静止状态能被修改吗？ | 完整性检查、参数化查询、HTTPS |
| **R**epudiation（抵赖） | 操作能被事后否认吗？ | 安全事件审计日志 |
| **I**nformation disclosure（信息泄露） | 数据会泄露吗？ | 加密、字段白名单、通用错误消息 |
| **D**enial of service（拒绝服务） | 能被压垮吗？ | 速率限制、输入大小上限、超时 |
| **E**levation of privilege（权限提升） | 用户能获得不应有的权限吗？ | 授权检查、最小权限 |

4. **在每个用例旁写出滥用案例。** 对每个功能，问"我如何滥用它？"——然后让那成为你的第一个测试。

如果你说不出信任边界，你就还没准备好加固它。这就是 OWASP **A04: Insecure Design**——大多数入侵始于设计，而非代码。

## OWASP 核心防御

### 注入（SQL、NoSQL、OS 命令）

使用参数化查询或 ORM。永远不要将用户输入拼接到查询中。对于 OS 命令，使用白名单参数验证。

### 认证缺陷

使用 bcrypt 哈希密码（salt rounds ≥ 12）。Session cookie：`httpOnly`、`secure`、`sameSite: lax`，24h 后过期。对认证端点进行速率限制（15 分钟内 10 次尝试）。

### 跨站脚本（XSS）

使用框架内置的输出编码。如果必须从不受信任的来源渲染原始标记，使用维护良好的清理库进行清理。永远不要将不受信任的数据直接传递给未转义的输出。

### 访问控制缺陷

在每个受保护端点上检查授权，而不仅是认证。用户只能访问自己的资源。管理员操作需要管理员角色验证。

### 安全配置错误

设置安全头。使用限制性指令配置 CSP。CORS 限制为已知来源——永远不要使用 `*` 带 credentials。

### 敏感数据暴露

使用清理函数从 API 响应中剥离敏感字段（`passwordHash`、`resetToken`）。所有密钥来自环境变量，永远不在代码中。

### 服务端请求伪造（SSRF）

任何服务端对用户影响的 URL 的获取都是攻击面（webhook、图片代理、链接预览）。白名单校验 scheme 和 host，拒绝私有/保留 IP，禁止重定向。注意：DNS 重绑定是 TOCTOU 风险——对于高风险面，通过过滤 Agent 获取。

完整 OWASP 速查表见 `references/security-checklist.md`。

## 依赖审计结果分级处置

```
依赖审计报告漏洞
├── 严重程度: Critical 或 High
│   ├── 生产中可达？→ 立即修复
│   └── 仅开发依赖/不可达？→ 尽快修复，非阻塞项
├── 严重程度: Moderate → 下一个发布周期修复
└── 严重程度: Low → 跟踪并在常规依赖更新时修复
```

如果推迟修复，记录原因并设定审查日期。

### 供应链安全管理

- **提交 lockfile**，在 CI 中使用干净安装——可重现构建，杜绝依赖悄然变更。
- **添加新依赖前审查**——维护情况、下载量、是否物有所值。每个依赖都是攻击面。
- **警惕 `postinstall` 脚本**和拼写仿冒攻击。

## 密钥管理

```
.env.example  → 已提交（带占位符的模板）
.env          → 不提交（包含真实密钥）

.gitignore 必须包含: .env, .env.local, .env.*.local, *.pem, *.key
```

**提交前检查:** `git diff --cached | grep -i "password\|secret\|api_key\|token"`

**如果密钥被提交了，立即轮换。** 仅仅删除或重写历史不够——假设一旦到达远程就被入侵。先撤销并重新签发密钥，再从历史中清除。

## 保护 AI / LLM 功能

如果你的应用调用 LLM，它继承了一个新的攻击面（OWASP Top 10 for LLM Applications 2025）：

- **将所有模型输出视为不信任的输入（LLM05）。** 永远不要将 LLM 输出传入代码执行、SQL、shell、未转义的输出或文件路径。像对待原始用户输入一样验证和编码。
- **假设提示词可被劫持（LLM01）。** 上下文窗口中的不信任文本可以携带指令。系统提示词不是安全边界；在代码中强制权限，不在提示词中。
- **不要让密钥和其他用户的数据进入提示词（LLM02/LLM07）。** 上下文中的任何内容都可以被回显。
- **约束工具权限（LLM06）。** 将工具范围限定到最小，对破坏性操作要求确认，验证每个工具参数。
- **限制消费（LLM10）。** 对 token、请求速率和循环/递归深度设置上限。
- **隔离检索数据（LLM08）。** 在 RAG 中，按租户分区 embedding，在索引之前验证文档。

## 警告信号

- 用户输入直接传递给查询、shell 命令或未转义输出
- 密钥在源代码或 commit 历史中
- API 端点没有认证或授权检查
- 具有已知严重漏洞的依赖
- LLM 输出被传入代码执行或未转义输出

## 验证

- [ ] 依赖审计显示无 Critical 或 High 漏洞
- [ ] 源代码或 git 历史中无密钥
- [ ] 所有用户输入在系统边界验证
- [ ] 每个受保护端点上检查了认证和授权
- [ ] 响应中存在安全头
- [ ] 错误响应不暴露内部详情
- [ ] 认证端点上活跃速率限制
- [ ] 服务端 URL 获取已针对白名单验证（无 SSRF）
- [ ] LLM/模型输出在使用前已验证和编码（如果存在 AI 功能）
