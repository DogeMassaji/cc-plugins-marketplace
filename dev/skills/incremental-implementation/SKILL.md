# 增量实现

## 概述

以垂直切片方式构建——实现一片、测试、验证、然后扩展。避免一次性实现整个功能。每个增量应该使系统处于可工作、可测试的状态。

## 何时使用

- 实现任何涉及多个文件的变更
- 从任务拆解中构建新功能
- 重构现有代码
- 任何你想要在测试之前写超过 ~100 行代码的时候

**何时不使用：** 单文件、单函数变更，范围已经最小。

## 增量循环

```
┌──────────────────────────────────────┐
│                                      │
│   实现 ──→ 测试 ──→ 验证 ──┐        │
│       ▲                   │        │
│       └───── 提交 ◄───────┘        │
│              │                     │
│              ▼                     │
│          下一个切片                  │
│                                      │
└──────────────────────────────────────┘
```

每个切片：

1. **实现** 最小的完整功能片段
2. **编译检查** — 运行编译/类型检查器，确保无编译错误和类型错误（如 TypeScript/Go/Rust 等有类型系统的语言，必须通过类型检查）
3. **测试** — 运行测试套件（如果没有测试就写一个）
4. **验证** — 确认切片按预期工作（测试通过、构建成功、手动检查）
5. **提交** — 用描述性 message 保存你的进度
6. **进入下一个切片** — 延续，不要重启

## 切片策略

### 纵向切片（推荐）

构建贯穿技术栈的一条完整路径：

```
切片 1: 创建任务（DB + API + 基本 UI）
    → 测试通过，用户可以通过 UI 创建任务

切片 2: 列出任务（查询 + API + UI）
    → 测试通过，用户可以查看他们的任务

切片 3: 编辑任务（更新 + API + UI）
    → 测试通过，用户可以修改任务

切片 4: 删除任务（删除 + API + UI + 确认）
    → 测试通过，完整 CRUD 完成
```

每个切片交付可工作的端到端功能。

### 契约优先切片

当服务端和客户端需要并行开发时：

```
切片 0: 定义 API 契约（类型、接口、spec）
切片 1a: 基于契约实现后端 + API 测试
切片 1b: 基于匹配契约的 mock 数据实现客户端
切片 2: 集成并端到端测试
```

### 风险优先切片

先处理最有风险或最不确定的部分：

```
切片 1: 证明 WebSocket 连接可行（最高风险）
切片 2: 在已验证的连接上构建实时任务更新
切片 3: 添加离线支持和重连
```

如果切片 1 失败了，你在投入切片 2 和 3 之前就发现了。

## 实现规则

### 规则 0：简洁优先

写任何代码之前，问自己："能工作的最简单方案是什么？"

```
简洁检查：
✗ 为一个通知使用带中间件管道的通用 EventBus
✓ 简单的函数调用

✗ 为两个类似组件使用抽象工厂模式
✓ 两个直接组件加共享工具

✗ 为三个表单使用配置驱动的表单构建器
✓ 三个表单组件
```

三行相似的代码好于一个过早的抽象。

### 规则 0.5：范围控制

只改任务要求的内容。

不要：
- "顺便清理"与你的变更相邻的代码
- 重构你没在修改的文件中的 import
- 删除你不完全理解的注释
- 添加 spec 中没有的功能因为"看起来有用"
- 给你只是在读的文件做语法现代化

如果你注意到任务范围之外值得改进的地方，记录下来——不要修复它：

```
注意到但未触及：
- src/utils/format 有一个未使用的 import（与本次任务无关）
- auth 中间件可以用更好的错误消息（单独的任务）
→ 需要我为这些创建任务吗？
```

### 规则 1：一次一件事

每个增量只改变一个逻辑事项。不要混在一起：

**错误：** 一个 commit 同时添加新组件、重构现有组件、更新构建配置。

**正确：** 三个独立的 commit——每个变更一个。

### 规则 2：保持可编译

每个增量之后，项目必须能构建且已有测试必须通过。不要在切片之间使代码库处于损坏状态。

### 规则 3：安全默认值

新代码应该默认为安全、保守的行为：

```
// 安全：默认禁用，主动选择启用
function createTask(data, options?):
  shouldNotify = options?.notify ?? false
```

### 规则 4：可回滚

每个增量应该可以独立回滚。增量变更（新文件、新函数）易于回滚。避免在一个 commit 中删除又替换——分开做。

## 增量检查清单

每个增量之后，验证：

- [ ] 变更做了一件事且完整地做了它
- [ ] 所有已有测试仍然通过
- [ ] 构建成功
- [ ] 类型检查通过（如适用）
- [ ] Lint 通过（如适用）
- [ ] 新功能按预期工作
- [ ] 变更已用描述性 message 提交

## 警告信号

- 超过 100 行没运行测试
- 单个增量中多个不相关变更
- "让我也快速加上这个"范围蔓延
- 增量之间构建或测试损坏
- 在第三个用例前就构建抽象

## 验证

完成任务的所有增量后：

- [ ] 每个增量都已单独测试并提交
- [ ] 编译/类型检查通过
- [ ] 完整测试套件通过
- [ ] 构建干净
- [ ] 功能按指定端到端工作
- [ ] 没有未提交的变更残留
