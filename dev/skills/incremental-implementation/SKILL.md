# 增量实现

## 概述

垂直切片构建——实现一片、测试、验证、扩展。避一次性实现整功能。每个增量使系统处可工作、可测试状态。

## 何时用

- 实现涉多文件变更
- 从任务拆解构建新功能
- 重构已有代码
- 测试前需要写 >100 行时

**不用时：** 单文件单函数变更，范围已最小。

## 增量循环

```
┌─────────────────────────────────────────┐
│  实现 → 测试 → 验证 → 提交 → 下一切片   │
└─────────────────────────────────────────┘
```

每个切片：
1. **实现** 最小完整功能片段
2. **编译检查** — 通过编译/类型检查
3. **测试** — 跑测试
4. **验证** — 确认按预期工作
5. **提交** — 描述性 message
6. **下一切片** — 延续，不重启

## 切片策略

### 纵向切片（推荐）

贯穿技术栈一条完整路径：

```
切片 1: 创建任务（DB + API + 基本 UI）→ 用户能用 UI 创建
切片 2: 列出任务（查询 + API + UI）→ 用户能查看
切片 3: 编辑任务（更新 + API + UI）→ 用户能修改
切片 4: 删除任务（删除 + API + UI + 确认）→ 完整 CRUD
```

每个切片交付可工作端到端功能。

### 契约优先

前后端需并行开发时：

```
切片 0: 定义 API 契约（类型、接口、spec）
切片 1a: 基于契约实现后端 + API 测试
切片 1b: 基于 mock 实现客户端
切片 2: 集成端到端测试
```

### 风险优先

先处理最不确定部分：

```
切片 1: 证明 WebSocket 可行（最高风险）
切片 2: 在已连接上构建实时更新
切片 3: 添加离线支持和重连
```

切片 1 失败则投入 2 和 3 前就发现。

## 规则

### 规则 0：简洁优先

写代码前问："能工作的最简单方案？"

```
✗ 通知用带中间件管道的通用 EventBus
✓ 简单函数调用

✗ 两个类似组件用抽象工厂
✓ 两个直接组件加共享工具

✗ 三个表单用配置驱动表单构建器
✓ 三个表单组件
```

三行相似代码好过一个过早抽象。

### 规则 0.5：范围控制

只改任务要求内容。

不要：
- "顺便清理"旁边代码
- 重构没在改的文件 import
- 删不完全理解的注释
- 加 spec 中没有的功能
- 给只是读的文件做语法现代化

注意到范围外的改进点，记录不修复：
```
注意到但未触及：
- src/utils/format 有未用 import（与本任务无关）
→ 需要为这些创建任务吗？
```

### 规则 1：一次一事

每个增量只改一个逻辑事项：

**错误：** 一 commit 同时加新组件、重构已有、更新构建配置。

**正确：** 三独立 commit——每变更一个。

### 规则 2：保持可编译

每个增量后项目必须能构建，已有测试通过。不使代码库处损坏状态。

### 规则 3：安全默认

新代码默认安全保守：

```js
function createTask(data, options?):
  shouldNotify = options?.notify ?? false
```

### 规则 4：可回滚

每个增量可独立回滚。增量变更（新文件、新函数）易回滚。避一 commit 删又替换——分开做。

## 增量检查清单

每个增量后验证：
- [ ] 做了一件事且完整
- [ ] 已有测试全通过
- [ ] 构建成功
- [ ] 类型检查通过
- [ ] Lint 通过
- [ ] 新功能按预期工作
- [ ] 已用描述性 message 提交

## 警告

- >100 行没跑测试
- 单增量多不相关变更
- "再快速加个这个"范围蔓延
- 增量间构建或测试损坏
- 第三个用例前就建抽象

## 验证

任务完成所有增量后：
- [ ] 每个增量已单独测试并提交
- [ ] 编译/类型检查通过
- [ ] 完整测试套件通过
- [ ] 构建干净
- [ ] 功能按指定端到端工作
- [ ] 无未提交变更残留
