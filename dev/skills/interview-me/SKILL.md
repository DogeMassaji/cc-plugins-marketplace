# 需求访谈

## 概述

人说出需求和真需求是两回事。要"dashboard"因这是约定说法，非因能解决问题。说"更快"但无量化指标。

捕捉差距最便宜时机在 plan/spec/code 存在前。开始构建后，切换成本真实，用户给错误东西合理化为"够好了"。

本 Skill 在造成损失前关掉差距。Interview-me 提取真实意图；意图仍模糊（"类似..."但无具体形态），则应用结构化发散/收敛思维生成和压力测试方案，再交接 `dev:spec-driven-development`。

## 何时用

- 需求至少缺一项：**谁**是用户、**为什么**需要、成功标准、硬性**约束**
- 请求约定俗成不具体（"构建 X"、"更快"）
- 倾向从未暴露假设开始
- 两合理价值观冲突（简洁 vs. 灵活、成本 vs. 速度），用户没指优先哪个
- 用户明确触发：/"interview me"、"拷问"、"确定吗？"、"检验想法"

**不用时：**

- 需求明确独 立（"重命名变量"、"修复拼写"）
- 用户明确要速度优先验证
- 纯信息请求（"X 怎么工作？"）
- 机械操作（重命名、格式化、移文件）
- 置信度 ≥95%

## 流程

### 第 1 步：提假设标置信度

问任何问题前，**一句话**写最佳解读 + 诚实的置信度（0–100%）：

```
假设：你想要在站会上答"做得怎么样"的方式，"dashboard"是最先想到的约定说法。
置信度：~30% — 缺：为谁做、"指标"指什么、成功标准
```

数字强制诚实。当你 <~70%，附简短原因——告诉用户访谈需揭示什么。

### 第 2 步：一次一问，附带猜测

```
Q: <一个聚焦问题>
GUESS: <假设 + 推理>
```

等用户反应再问下一个。

**一次一个原因：**
- 多假设埋列表中用户无法反应
- 批量鼓励略读
- 第三个问题往往取决第一个答案
- 用户精力有限

**附带猜测原因：**
- 用户对错猜测反应比从零生成更快
- 绑定到可明显出错的假设，保持诚实
- 暴露*你*的假设

风险：礼貌用户会同意猜测以示配合。通过显 露出错意愿缓解，偶尔猜个预期被反驳方向。

### 第 3 步：听"想要 vs. 应该想要"

最危险答案是用户说深思熟虑答案*听起来像*什么，非真想要。注意：

- 模式匹配最佳实践话（"可扩展"、"整洁架构"）但无细节
- 遵从惯例（"大多数应用那样做"）
- 短语如"大概应该…"、"好的工程实践说…"
- 用流行词当目标

当听到，问：
> *"不需要向人解释理由，你真正想要什么？"*

这个问题往往比前五个有用。

### 第 4 步：用用户语言重申

置信度很高时，写回你认为用户想要什么。5–8 行，用他们语言，结构化以便逐行确认：

```
以下是我现认为你想要的：

- 成果：     <一行>
- 用户：     <一行>
- 为什么现在：<一行>
- 成功标准：  <一行>
- 约束：     <一行>
- 排除范围：  <一行>

是 / 否 / 需调整？
```

"排除范围"不可协商。一半对齐失败是关于什么*不*该建的无声分歧。

### 第 5 步：确认——明确"是"

门槛是明确"是"。以下**不是**：

- "你觉得怎么好就怎么做"→ 授权，非确认。给两个具体选项重选
- "听起来不错"→ 模糊。追问"有要调的吗？"
- "当然，开始吧"→ 通常礼貌退出。同样追问
- 沉默后"好吧开始"→ 用户放弃访谈。问是否错过什么

纠正则纳入并重述。循环到明确"是"。

**95% 停止条件：**

> *我能预测用户对下三个问题的反应吗？*

是则完成，产出重述。否则继续。

如多轮仍无法预测，停止说："已问 X 个问题仍无法预测。一些基础性东西缺失。退一步重新梳理？"

### 第 6 步：精炼模糊想法

确认意图为"类似 X 但不知形态"时，用发散→收敛生成和缩窄选项。

**此步适用：**
- 用户说"头脑风暴"、"精炼"、"给选项"、"压力测试"
- 确认意图有目标无方法
- 多合理架构且 trade-off 存在

**发散——生成选项（2–4 个变体）：**
- **它是什么**：一句话
- **最适合**：优化给谁
- **Trade-off**：具体代价（延迟、复杂度、依赖、学习曲线）
- **何时不选**：反信号

**收敛——压力测试缩窄：**
- **匹配度**：匹配实际成果？非约定答案？
- **约束**：违反硬性限制？
- **简洁性**：更简单版本仍满足？
- **风险**：最坏失败模式？

推前 1–2 个选项，附推荐和推理。用户选，不替他们选。

**示例：**

```
意图确认：个人实验追踪器。三种形态：

选项 A — Markdown + git hook
  → 最适合：单人，零配置，版本受控
  → Trade-off：无搜索，无可视化
  → 反信号：需图表或共享

选项 B — SQLite + CLI
  → 最适合：可查询、可脚本、私密
  → Trade-off：你自己建 UI
  → 反信号：想要 web 界面

选项 C — 轻量 Web（SQLite + htmx）
  → 最适合：web 访问，可视化，仍本地
  → Trade-off：最多配置
  → 反信号：今天要完成

推荐：A 或 B。核心需求是列表非 dashboard。
```

## 输出

一份**确认意图陈述**：第 4 步重述 + 第 5 步明确"是"。Spec/plan/task list 是下游产物。

如用户希望持久化，提议存 `docs/intent/[topic].md`。仅确认后保存。

## 示例

**未用 interview-me：**

```
用户： 建我们指标的 dashboard
Agent：<开始提议图表库和布局>
```

Agent 已假设：为谁、什么指标、"我们"什么意思、成功样子。每个假设都是以后可能说"不是那意思"的地方。

**用了：**

```
用户：建我们指标的 dashboard

Agent：
假设：想要站会上答"做得怎么样"，"dashboard"只是约定说法。
置信度：~30% — 缺：为谁、什么"指标"、成功样子

Q:    "做得怎么样"，谁在问——你、工程团队站会、还是上级？
GUESS：工程团队站会。"我们"通常这样用。

用户：其实为我自己。总不知哪些实验在跑，早期信号什么样。

Agent：
更新解读：非团队 dashboard。个人实验追踪器。置信度：~60%

Q:    问题是不知哪些实验存在，还是无法一处看结果？
GUESS：第二个。实验列表在各处，结果分散五个工具。

用户：其实是第一个。根本无列表。分散在文档里。

Agent：<继续>
```

两问后，Agent 已发现实际需求非"dashboard"，是"列表"。不同产物、范围、工作量。

## 关联 Skill

- **`dev:spec-driven-development`**：下游。意图具体且方案确定后交接
- **`dev:planning-and-task-breakdown`**：隔一 hop 下游（spec 之后）

## 警告

- 单条消息三问以上（批量非访谈）
- 问题不带假设（问卷非承诺）
- 接受"你觉得怎么好就怎么做"当终答案
- 三轮后置信度未明显升（问错问题）
- 用户确认前出 spec/plan/task list

## 验证

应用后：
- [ ] 首轮陈述明确假设和置信度
- [ ] 问题一次一个，附猜测
- [ ] 具体重述（成果/用户/原因/成功/约束/排除）获用户确认
- [ ] 能预测对下三问题反应
- [ ] 交接到下游基于确认意图，非原始模糊需求
