# 文档归档整理器

按照标准化目录结构归档项目文档，保持文档可发现、可追溯、可维护。

## 目录结构规范

```
docs/
├── 01-项目概述/        # 项目级文档：构思、架构、TODO
├── 02-技术验证/        # 技术验证、实验、调研
├── 03-设计决策/        # 架构决策记录 (ADR)、设计取舍
├── ...                # 按需新增编号分类
└── 99-已废弃/          # 已废弃的方案、旧文档
```

**编号规则：**
- `01`–`98`：活跃分类，按重要性/时间线排列
- `99`：固定为"已废弃"，所有废弃文档移入此处
- 新分类取当前最大编号 +1

**分类命名：** `NN-中文分类名`，简洁 2-4 字。

## 文件格式规范

### 文件命名

中文描述性文件名，不加编号前缀（目录已有编号）。例：
- `项目构思.md`
- `CDP事件isTrusted验证.md`
- `人类化鼠标轨迹模块.md`

### 文档头部

```markdown
# 文档标题

> 日期: YYYY-MM-DD | 状态: 草稿/已验证/已废弃
```

状态枚举：`草稿` → `已验证` → `已废弃`。

### 正文结构

```markdown
---

## 一、概述/背景

...

## 二、核心内容

...

---

## 验证/结论

- [ ] 检查项 1
- [ ] 检查项 2
```

- 用 `---` 分隔大章节
- 中文数字编号大标题
- 表格用于对比、技术选型
- 代码块标注语言
- 末尾附验证检查清单

### 废弃文档格式

废弃时在头部增加废因字段，然后移入 `99-已废弃/`：

```markdown
# 文档标题

> **废弃日期**: YYYY-MM-DD
> **废弃原因**: 一句话说明为何废弃
> **替代方案**: 见 `docs/NN-分类/新文档.md`
```

## TODO.md 格式

每个活跃分类目录下可有一个 `TODO.md`：

```markdown
# TODO

## YYYY-MM-DD

- [x] 已完成项 (commit_hash)
- [ ] 待办项
```

- 日期倒序，最新在上
- 已完成项标注 commit hash
- 每天工作结束或切换任务时更新

## 工作流程

### Step 1: 确定归档位置

1. 读取 `docs/` 目录树，了解现有分类
2. 判断新文档属于哪个现有分类
3. 如无匹配分类，创建新编号目录
4. 如涉及废弃旧方案，同步规划废弃文档迁移

### Step 2: 创建/更新文档

1. 按文件格式规范写文档
2. 如有代码引用，标注相对路径和行号
3. 如有外部参考，用 markdown 链接
4. 涉及技术验证的，附验证脚本路径和结果

### Step 3: 更新 TODO

1. 若任务与文档内容相关，更新对应分类下的 `TODO.md`
2. 已完成项标注 commit hash
3. 新增待办项追加到当天日期段

### Step 4: 输出归档摘要

```
文档归档完成。
- 新增: docs/02-技术验证/xxx.md
- 更新: docs/01-项目概述/TODO.md
- 废弃: docs/99-已废弃/旧方案.md
```

## 硬性要求

1. **编号目录** — 所有文档必须在编号目录下，不在 `docs/` 根目录散放
2. **中文文件名** — 文件名用中文描述，不拼音不英文（代码/技术术语除外）
3. **头部元数据** — 每个文档必须有 `# 标题` 和 `> 日期 | 状态` 行
4. **废文入 99** — 废弃文档必须移入 `99-已废弃/`，不在原目录残留
5. **TODO 同步** — 文档内容涉及的任务必须在 `TODO.md` 有对应条目

## 硬性禁止

1. **禁止英文文件名** — 除代码标识符外，文件名用中文
2. **禁止无头文档** — 不允许缺少日期和状态的文档
3. **禁止废文原地留** — 废弃文档不能留在活跃分类目录
4. **禁止空目录** — 新建分类目录必须至少包含一个 `.md` 文件
5. **禁止单独 md 散放** — 文档必须归属到某个编号分类下，不在 `docs/` 根目录

## 验证

完成归档后确认：
- [ ] 所有文档在编号目录下
- [ ] 每个文档有日期和状态头
- [ ] 文件名用中文
- [ ] 废弃文档在 `99-已废弃/` 且标注废因
- [ ] TODO.md 记录已同步
