# 计划与任务拆分

## 概述

将工作拆为小、可验证任务，附验收标准。好任务拆分是 Agent 可靠工作和混乱产出间区别。每个任务应小到可在单次 session 中实现、测试、验证。

## 何时用

- 有 spec 需拆为实施单元
- 任务感觉太大或模糊难开始
- 需多 Agent/session 并行
- 需向人传达范围
- 实现顺序不明显

**不用时：** 单文件变更，或 spec 已有良好定义任务。

## 流程

### 第 1 步：Plan Mode

写代码前只读操作：
- 读 spec 和相关代码
- 识别模式和约定
- 绘依赖关系
- 标风险和未知

**规划期不写代码。** 输出是计划文档，非实现。

### 第 2 步：识别依赖图

```
DB Schema → API 模型/类型 → API 端点 → 客户端调用 → UI 组件
                     ↘ 验证逻辑
```

实现自底向上：先建基础。

### 第 3 步：纵向切片

非先建所有 DB、所有 API、所有 UI——一次建一条完整功能路径：

**错（横向）：**
```
任务 1: 建整个 DB Schema
任务 2: 建所有 API
任务 3: 建所有 UI
任务 4: 全连起来
```

**对（纵向）：**
```
任务 1: 用户注册（Schema + API + UI）
任务 2: 用户登录（认证 Schema + API + UI）
任务 3: 用户创任务（任务 Schema + API + UI）
任务 4: 用户看任务列表（查询 + API + UI）
```

每个纵向切片交付可工作、可测试功能。

### 第 4 步：写任务

每个任务：

```markdown
## 任务 [N]: [标题]

**描述：** 一段话说明任务完成什么。

**验收标准：**
- [ ] 具体可测条件

**验证：**
- [ ] 测试通过
- [ ] 构建成功
- [ ] 手动检查

**依赖：** [任务编号或"无"]

**涉及文件：**
- `src/path/to/file.ts`

**预估：** [小: 1-2 文件 | 中: 3-5 | 大: 5+]
```

### 第 5 步：排序与检查点

1. 依赖满足
2. 每个任务使系统保持工作状态
3. 每 2–3 任务后验检查点
4. 高风险靠前

```markdown
## 检查点: 任务 1-3 后
- [ ] 所有测试通过
- [ ] 构建无错误
- [ ] 核心流程端到端工作
- [ ] 与人类审查
```

## 计划文档模板

```markdown
# 实现计划: [功能名]

## 概述

## 架构决策
- [决策 1 及理由]

## 任务列表

### 阶段 1: 基础
- [ ] 任务 1
- [ ] 任务 2

### 检查点: 基础

### 阶段 2: 核心
- [ ] 任务 3

### 检查点: 核心

### 阶段 3: 打磨
- [ ] 任务 4

### 检查点: 完成
- [ ] 所有验收标准满足
- [ ] 准备好审查

## 风险与缓解
| 风险 | 影响 | 缓解 |

## 待定问题
```

## 警告

- 无书面任务列表就开实现
- 任务只写"实现功能"无验收标准
- 任务间无检查点
- 任务涉 >8 文件（需进一步拆）

## 验证

开始实现前确认：
- [ ] 每个任务有验收标准
- [ ] 每个任务有验证步骤
- [ ] 依赖已识别并正确排序
- [ ] 无任务涉 >~5 文件
- [ ] 主要阶段间有检查点
- [ ] 用户已审查批准计划
