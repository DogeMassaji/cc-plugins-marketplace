# Spec 驱动开发

## 概述

在编写任何代码之前编写结构化的 Spec 说明。Spec 是你和人类工程师之间的共享真理——它定义了我们正在构建什么、为什么构建、以及如何知道它完成了。没有 spec 的代码就是猜测。

## 何时使用

- 开始一个新项目或功能
- 需求模糊或不完整
- 变更涉及多个文件或模块
- 即将做出架构决策
- 任务的实现预估超过 30 分钟

**何时不使用：** 单行修复、拼写纠正、或者需求明确的独立变更。

## 分阶段门控流程

Spec 驱动开发有四个阶段。在当前阶段验证通过之前，不要进入下一阶段。

```
SPECIFY ──→ PLAN ──→ TASKS ──→ IMPLEMENT
   │          │        │          │
   ▼          ▼        ▼          ▼
 人工审核    人工审核   人工审核    人工审核
```

### 阶段 1：Specify

从高层愿景开始。向人类提问澄清，直到需求具体。

**立即暴露假设。** 在编写任何 spec 内容之前，列出你的假设：

```
我当前的假设：
1. 这是一个 Web 应用（而非原生移动端）
2. 认证使用基于 session 的 cookie（而非 JWT）
3. 数据库是 PostgreSQL（基于现有 Prisma schema）
4. 目标为现代运行时环境
→ 现在纠正我，否则我将基于这些假设继续。
```

不要默默填充模糊的需求。Spec 的全部目的就是在代码编写*之前*揭示误解——假设是最危险的误解形式。

**编写覆盖六大核心领域的 spec 文档：**

1. **目标（Objective）** — 我们正在构建什么，为什么？用户是谁？成功的样子是什么？

2. **命令（Commands）** — 完整的可执行命令及标志，不仅工具名称。
   ```
   Build: <项目构建命令>
   Test: <项目测试命令>
   Lint: <项目 lint 命令>
   Dev: <项目开发命令>
   ```

3. **项目结构（Project Structure）** — 源代码在哪里，测试在哪里，文档在哪里。
   ```
   src/           → 应用源码
   src/components → UI 组件
   src/lib        → 共享工具
   tests/         → 单元和集成测试
   e2e/           → 端到端测试
   docs/          → 文档
   ```

4. **代码风格（Code Style）** — 一段真实代码示例展示你的风格，胜过三段落描述它。包含命名约定、格式化规则、好输出的示例。

5. **测试策略（Testing Strategy）** — 什么框架、测试放在哪里、覆盖率期望、哪些测试级别用于哪些关注点。

6. **边界（Boundaries）** — 三层体系：
   - **Always do:** 提交前运行测试、遵循命名约定、验证输入
   - **Ask first:** 数据库 schema 变更、添加依赖、修改 CI 配置
   - **Never do:** 提交密钥、编辑 vendor 目录、未经批准删除失败测试

**Spec 模板：**

```markdown
# Spec: [项目/功能名称]

## 目标
[我们正在构建什么，为什么。用户故事或验收标准。]

## 技术栈
[框架、语言、关键依赖及版本]

## 命令
[Build、test、lint、dev——完整命令]

## 项目结构
[目录布局及描述]

## 代码风格
[示例片段 + 关键约定]

## 测试策略
[框架、测试位置、覆盖率要求、测试级别]

## 边界
- Always: [...]
- Ask first: [...]
- Never: [...]

## 成功标准
[我们如何知道这个完成了——具体的、可测试的条件]

## 待定问题
[任何未解决的、需要人工输入的问题]
```

**将指令重构为成功标准。** 当收到模糊需求时，将其转化为具体条件：

```
需求："让 dashboard 更快"

重构后的成功标准：
- Dashboard 首次内容绘制 < 2.5s（4G 连接）
- 初始数据加载在 500ms 内完成
- 加载期间无布局偏移（CLS < 0.1）
→ 这些是正确的目标吗？
```

这使你能够循环、重试、解决问题直至明确目标，而非猜测"更快"意味着什么。

### 阶段 2：Plan

基于已验证的 spec，生成技术实现计划：

1. 识别主要组件及其依赖关系
2. 确定实现顺序（什么必须先构建）
3. 标注风险和缓解策略
4. 识别什么可以并行 vs. 什么必须串行
5. 定义阶段之间的验证检查点

计划应该是可审查的：人类应该能够阅读它并说"是的，这是正确的方法"或"不，改一下 X。"

### 阶段 3：Tasks

将计划分解为离散的、可实施的任务：

- 每个任务应该能在单次聚焦 session 中完成
- 每个任务有明确的验收标准
- 每个任务包含验证步骤（测试、构建、手动检查）
- 任务按依赖排序，而非按感知重要性
- 没有任务应该涉及超过 ~5 个文件

**任务模板：**
```markdown
- [ ] 任务: [描述]
  - 验收标准: [完成时必须为真的条件]
  - 验证方式: [如何确认——测试命令、构建、手动检查]
  - 涉及文件: [哪些文件将被改动]
```

### 阶段 4：Implement

一次一个任务，遵循 `skills/incremental-implementation/SKILL.md`（`dev:incremental-implementation`）执行。

## 保持 Spec 更新

Spec 是活文档，不是一次性产物：

- **当决策变更时更新** — 如果你发现数据模型需要改变，先更新 spec，再实现。
- **当范围变化时更新** — 新增或移除的功能应反映在 spec 中。
- **提交 spec** — Spec 应与代码一起放在版本控制中。
- **在 PR 中引用 spec** — 链接到每个 PR 实现的 spec 部分。

## 警告信号

- 没有任何书面需求就开始写代码
- 在明确"完成"的含义之前就"直接开始构建"
- 实现 spec 或任务列表中未提及的功能

## 验证

进入实现阶段之前，确认：

- [ ] Spec 覆盖了全部六大核心领域
- [ ] 人类已审查并批准了 spec
- [ ] 成功标准是具体且可测试的
- [ ] 边界（Always/Ask First/Never）已定义
- [ ] Spec 已保存到仓库中的文件
